## Phase 20 Plugin Runtime Foundation Planning

设计结论：

- 插件是安装/启停/诊断/contribution package，不是新的运行时真相源。
- `Runtime.Snapshot` 仍是长期进程世界观；插件贡献必须进入同一个 snapshot 后再被 Gateway / Runner / Workbench / ControlPlane 消费。
- 现有 `Channel.Catalog`、`ProviderRegistry`、`Tool.Registry`、`Skills.Catalog`、`Workbench.Store` 不是要被替换，而是要逐步改成 plugin contribution 的 consumer。
- Phase 20 分三阶段：Stage 1 只做 plugin host 抽象和 inventory；Stage 2 把现有 catalog 接到 plugin contributions；Stage 3 物理迁移 Feishu/Discord/provider/tool/builtin skill 等非核心能力。
- Review 后补硬了几条防第二主线约束：
  - `Runtime.Watcher` 必须扫描 plugin manifest，manifest 改动触发 reload、改变 `plugins.hash`，并让 Gateway/Workbench/runtime subscribers 看到同一个 runtime version。
  - `Config` 保留 raw provider type，provider availability 由 plugin-derived provider catalog 校验；disabled/unknown provider 不得静默变成 `:openai`。
  - `Tool.Registry` definitions 和 execution map 必须同源，禁用 tool plugin 后 `execute/3` 也返回 unknown tool。
  - Phase 20 冻结 contribution kind 收窄为 channels/providers/tools/skills/commands；Workbench app/view/subagent 只作为 deferred design，等 consumer 和 tests 一起落地。
  - command handler 在 manifest 中是 string，只能绑定到 bounded core/builtin handler table；workspace/project command 不提供 executable handler。

## Phase 18D Notes App CORS Follow-up

问题：

- Notes app iframe 使用 `sandbox="allow-scripts"`，因此 iframe document 是 opaque origin。
- Vite/ES module assets 从 `/app-assets/notes/dist/assets/*` 加载时会走 CORS。
- Workbench asset response 之前没有 CORS header，浏览器表现为静态资源 200 但 module/CSS 被 Same Origin Policy 拦截，页面空白。

实现：

- `Nex.Agent.Interface.Workbench.Server` 的 `{:asset, ...}` response 增加 app asset 专用 headers：
  - `Access-Control-Allow-Origin: *`
  - `Access-Control-Allow-Methods: GET, OPTIONS`
  - `Cross-Origin-Resource-Policy: cross-origin`
- 没有改 iframe sandbox 为 `allow-same-origin`，避免削弱 iframe 隔离。
- 没有把 Workbench JSON/API bridge CORS 放宽；修复只作用于静态 app assets。
- `server_test` 覆盖 `/app-assets/:id/*` headers。
