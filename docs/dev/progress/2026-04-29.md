## Phase 18D Notes App CORS Follow-up

问题：

- Notes app iframe 使用 `sandbox="allow-scripts"`，因此 iframe document 是 opaque origin。
- Vite/ES module assets 从 `/app-assets/notes/dist/assets/*` 加载时会走 CORS。
- Workbench asset response 之前没有 CORS header，浏览器表现为静态资源 200 但 module/CSS 被 Same Origin Policy 拦截，页面空白。

实现：

- `Nex.Agent.Workbench.Server` 的 `{:asset, ...}` response 增加 app asset 专用 headers：
  - `Access-Control-Allow-Origin: *`
  - `Access-Control-Allow-Methods: GET, OPTIONS`
  - `Cross-Origin-Resource-Policy: cross-origin`
- 没有改 iframe sandbox 为 `allow-same-origin`，避免削弱 iframe 隔离。
- 没有把 Workbench JSON/API bridge CORS 放宽；修复只作用于静态 app assets。
- `server_test` 覆盖 `/app-assets/:id/*` headers。

验证：

```bash
/opt/homebrew/bin/mix test test/nex/agent/workbench/server_test.exs test/nex/agent/workbench/assets_test.exs
/opt/homebrew/bin/mix compile --warnings-as-errors
```

结果：

- Workbench server/assets tests: 18 tests, 0 failures
- `compile --warnings-as-errors`: passed

现场注意：

- 尝试重启真实 gateway 以应用新代码时，Workbench 一度监听 `127.0.0.1:50051`，但 Feishu WS endpoint 请求随后失败为 `%Req.TransportError{reason: :nxdomain}`，gateway task 退出。
- 这次退出不是 CORS 修复导致；需要在外部网络/DNS 可用时重新启动 gateway 再刷新 Workbench。

## Phase 18D Workbench App Chrome Follow-up

实现：

- Workbench app manifest 新增 `chrome.topbar` contract：
  - 默认 `%{"topbar" => "auto"}`。
  - 允许 `"auto"` 或 `"hidden"`。
