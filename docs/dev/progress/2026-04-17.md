## Feishu 流式卡片两个用户可见 bug 修复

### 问题 1：上一个卡片打字动画不停

- 根因：卡片轮转时只发了 dummy update（同内容 +1 sequence），没有调飞书 `settings` API 关闭 `streaming_mode`。飞书客户端在 `streaming_mode: true` 期间会持续显示打字光标。
- 修复：
  - `CardBuilder` 创建卡片时加上 `streaming_config`（`print_strategy: "fast"`, `print_step: 2`, `print_frequency_ms: 50`）
  - `Feishu` 新增 `close_streaming_mode/1` API，调飞书 `PATCH /cardkit/v1/cards/:card_id/settings` 把 `streaming_mode` 设为 `false`
  - 卡片轮转和 finish 时都调 `close_streaming_mode`

### 问题 2：`<newmsg/>` 原样显示在卡片内容里

- 根因：LLM 在 markdown 示例中输出了嵌套的代码围栏（连续两个 ` ``` `），导致 `in_code_block?` 被错误 toggle 回 `true`，之后所有 `<newmsg/>` 都被跳过。
- 诊断手段：在 `do_consume` 中加了 `code_fence_toggle` 日志，抓到了 `false->true rest_preview="\n\n<newmsg/>\n\n# 第..."` 这条关键日志。
- 初始修复：把 `<newmsg/>` 检测提到 `in_code_block?` 之前。
- 最终修复：见下方架构简化。

## Feishu StreamConverter 架构简化

带着 phase 4/5/7 的架构意图重新 review 后，发现初始修复只是打补丁，整个逐字符状态机就不该存在。

### 删掉的复杂度

- 逐 grapheme 的 `do_consume` 状态机
- `in_code_block?` 状态追踪
- `current_line` / `separator_line_start?` / `line_prefix_for_segment` / `strip_separator_indent` 行首检测
- `code_fence` toggle 逻辑
- `defer_new_message?` 双路径（`push_text` defer vs `finish` 不 defer）
- `force_flush_active_card_animation` dummy update
- `token_prefixes` 枚举 `<newmsg/>` 和 ` ``` ` 的所有前缀

### 简化后的实现

- `consume`：按 `\n[ \t]*<newmsg/>[ \t]*\n` regex split（`parts: 2`），递归处理每段
- `split_processable`：只检查数据末尾是否有 `\n<newmsg/>` 的不完整前缀，不再关心 ` ``` `
- 遇到 `<newmsg/>` 立即切卡片，不 defer
- 结束打字动画只调 `close_streaming_mode`，不发 dummy update

### 核心判断

`<newmsg/>` 是我们自己注入 system prompt 的控制标记，LLM 不会在代码块内容中独占一行写它。不需要维护 `in_code_block?` 来保护 `<newmsg/>` 不被误匹配。

### 代码量

简化前 ~407 行 → 简化后 ~250 行。删掉 9 个内部函数。

## Discord 接入

### 已完成

1. **Auto-thread**：channel 中 @Bot → 自动用 `POST /channels/{channel_id}/messages/{message_id}/threads` 创建 thread → 回复进 thread。用消息前 80 字符做 thread name。
2. **Thread 免 @**：通过缓存 `known_threads`（MapSet）判断消息是否来自 thread。缓存来源：GUILD_CREATE 事件的 `threads` 数组 + THREAD_CREATE 事件 + bot 自己创建的 thread。和 discord.py 的 `guild._threads` 缓存机制一致。
3. **Discord StreamConverter**：和飞书 `StreamConverter` 同接口（`start/push_text/finish/fail`），内部管 `<newmsg/>` regex split + 2000 字符自动分段 + edit_message。
4. **InboundWorker 接入**：Discord streaming 改成和飞书一样的 converter + 1s 节流模式。删掉旧的 `discord_stream_state` / `flush_discord_stream` / `update_or_create_discord_message` / `send_discord_new_message` / `send_discord_segments` 共 5 个函数。
5. **System prompt**：`ContextBuilder` 为 Discord 注入了专属 IR guidance（支持/不支持的 markdown 语法、`<newmsg/>` 使用说明）。
6. **Gateway intents**：33281 已包含 GUILDS（bit 0），支持 GUILD_CREATE / THREAD_CREATE / THREAD_DELETE 事件。

### 已知问题待修

1. **飞书 `close_streaming_mode` 返回 404**：`PATCH /cardkit/v1/cards/:card_id/settings` 返回 `404 page not found`。可能是 API 路径不对或卡片已过期。需要对照飞书 CardKit 文档确认正确的 settings 接口路径和请求格式。当前 fallback 是忽略错误继续运行，不影响主流程但打字动画可能不会干净停止。
2. **Finch 连接池耗尽**：并发为 1 时仍出现 `excess queuing for connections` 错误。不是 pool size 问题——某些 LLM streaming 连接可能没被正确释放。需要排查 `ReqLLM` streaming 的连接生命周期：
   - 是否有 streaming response body 没被完整消费就丢弃的情况？
   - Finch 对未消费完的 streaming response 是否会持有连接直到超时？
   - `ReqLLM` 的 stream rescue 路径是否正确关闭了底层连接？
   - 建议给 Finch 连接加 acquire/release 日志，追踪哪个调用拿了连接不还。
3. **LLM 空返回**：模型 API 偶发返回空 `final_content`。不是代码 bug，但用户体验差（bot 沉默）。可以考虑在空返回时发一条 fallback 消息如"抱歉，我这次没能生成回复，请再试一次"。
