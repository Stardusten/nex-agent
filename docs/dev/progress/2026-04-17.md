# 2026-04-17

## Feishu Streaming Investigation

- Traced Feishu streaming `newmsg` behavior end-to-end with runtime logs on `InboundWorker`, `Feishu.StreamConverter`, and Feishu card API calls.
- Confirmed the original user-visible overlap was not a parsing bug: old-card update and next-card open could land close enough that Feishu still showed the previous card's streaming animation while the next card was already visible.
- Reproduced and fixed a regression where `<newmsg/>` split across flush boundaries could stop being recognized.
- Verified the current runtime again recognizes subsequent `<newmsg/>` boundaries after partial-token flushes.

## Dummy Update Result

- Implemented a `dummy update` experiment for Feishu CardKit streaming cards:
  - finish the old card with its real final content update
  - send the same content again with the next `sequence`
  - rotate and open the next card only after that dummy update returns
- Verified from logs that every investigated `newmsg` boundary now emits the dummy update before rotation/open.
- User-visible outcome still did not reliably force Feishu to terminate the old card's typing animation before showing the next card.
- Current conclusion from runtime evidence: the remaining overlap appears to be Feishu client rendering behavior rather than a missing server-side update.

## Official Docs Follow-Up

- Checked the Feishu CardKit streaming update overview page the user pointed to.
- The page indicates Feishu has a formal streaming-card model, but the attempted `/settings` close-streaming request used during this investigation returned `404` in runtime verification and was removed.
- The repo is currently left on the simpler verified path:
  - stable `newmsg` splitting
  - optional dummy update before rotation
  - no unsupported close-streaming request
