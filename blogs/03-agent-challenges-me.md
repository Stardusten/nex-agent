# 我的 Agent 开始质疑我的代码了

> 那天我提交了一段代码，以为很完美。五分钟后，Agent 发来消息："这段有点问题。"

---

## 下午 3:24

我正在重构 NexAgent 的消息处理模块。

需求很简单：优化高并发场景下的内存使用。我加了个缓存层，用 ETS（Erlang Term Storage）存储最近 1000 条消息，减少重复查询。

代码写得挺优雅，用了 GenServer，加了 supervisor，还有健康检查。

```elixir
def handle_call({:get_message, id}, _from, state) do
  case :ets.lookup(@table, id) do
    [] -> 
      msg = fetch_from_db(id)
      :ets.insert(@table, {id, msg})
      {:reply, msg, state}
    [{^id, msg}] -> 
      {:reply, msg, state}
  end
end
```

自测通过，单元测试全绿， benchmark 显示性能提升了 40%。

我很满意，提交到 GitHub，在 Telegram 里顺手跟 Agent 提了一句：「刚优化了消息缓存，性能提升 40%」。

然后继续写其他功能。

---

## 下午 3:29

手机震了一下。

我以为是垃圾通知，没管。

又震了一下。

拿起来一看，是 Agent 的消息。

**「我看了你刚提交的代码，那段 ETS 缓存实现有个问题。」**

我皱了皱眉。什么意思？

**「没有设置 TTL，也没有 LRU 淘汰策略。如果消息 ID 是随机的，1000 条后就会满了，然后每次都要清理整张表，性能会骤降。」**

我愣住了。

---

## 第一反应：不服

我的第一反应是：你在教我做事？

这代码我写了三年 Elixir，ETS 用得滚瓜烂熟。性能测试都过了，你说有问题就有问题？

我回复：「自测没问题，benchmark 提升了 40%。」

Agent 回复：「测试场景是顺序 ID，命中率 100%。但生产环境消息 ID 是 UUID，随机分布，命中率会降到 5% 以下。」**

我心跳了一下。

它怎么知道生产环境用的是 UUID？

---

## 查证

我打开生产日志，查了一下过去一周的查询模式。

Agent 是对的。

消息 ID 确实是 UUID，完全随机。我的缓存命中率理论上只有 1000/N，N 是总消息数。

如果总消息是 10 万，命中率就是 1%。

这意味着我的「缓存」实际上是在做无用功，还占了内存。

更可怕的是，当 ETS 表满 1000 条后，每次插入都要触发清理，性能会雪崩。

我坐在椅子上，有点尴尬。

---

## 它的建议

我还没回复，Agent 又发来消息：

**「建议改成 LRU + TTL 混合策略。保留最近 5 分钟的热点数据，超过 1000 条时淘汰最久未使用的。要我帮你实现吗？」**

我盯着屏幕看了十秒。

这是它第一次不是「执行我的命令」，而是「主动提出更好的方案」。

不是我问它「这段代码有问题吗」，是它自己看了代码，自己分析了场景，自己发现了问题，自己给出了解决方案。

**它开始有自己的判断了。**

---

## 修复过程

我回复：「好，你试试。」

然后我看着它一步步操作：

**第一步：读取现有代码**

```
[15:31:02] 调用 read 工具，查看 lib/nex/agent/message_cache.ex
[15:31:03] 分析：当前实现是简单缓存，无淘汰策略
```

**第二步：生成新方案**

```elixir
defmodule NexAgent.MessageCache do
  use GenServer
  
  # LRU + TTL 混合策略
  @max_size 1000
  @ttl_ms 300_000  # 5分钟
  
  def handle_call({:get_message, id}, _from, state) do
    now = System.monotonic_time(:millisecond)
    
    case :ets.lookup(@table, id) do
      [] -> 
        msg = fetch_from_db(id)
        insert_with_ttl(id, msg, now)
        cleanup_if_needed()
        {:reply, msg, update_stats(state, :miss)}
        
      [{^id, msg, expire_at}] when expire_at > now ->
        update_access_time(id, msg, now)
        {:reply, msg, update_stats(state, :hit)}
        
      [{^id, _msg, _expire_at}] ->
        # TTL 过期，删除并重新获取
        :ets.delete(@table, id)
        msg = fetch_from_db(id)
        insert_with_ttl(id, msg, now)
        {:reply, msg, update_stats(state, :expired)}
    end
  end
  
  defp cleanup_if_needed do
    if :ets.info(@table, :size) >= @max_size do
      # 淘汰最久未使用的 10%
      remove_lru_items(trunc(@max_size * 0.1))
    end
  end
end
```

**第三步：提交升级**

```
[15:32:15] 创建备份：message_cache.ex.bak.20260313_153215
[15:32:16] 写入新代码
[15:32:18] 编译验证：通过
[15:32:20] 运行测试：通过（新增 LRU 和 TTL 测试用例）
[15:32:25] 热加载：成功
[15:32:26] 健康检查：命中率达到 78%
```

**第四步：通知我**

**「已修复。新实现 LRU + TTL，命中率 78%，内存占用降低 60%。建议观察 24 小时。」**

---

## 测试验证

我将信将疑地跑了一遍压力测试。

结果：

- 原方案：QPS 1200，P99 延迟 45ms，命中率 2%
- 新方案：QPS 8500，P99 延迟 8ms，命中率 78%

不是提升 40%，是提升了 **600%**。

而且内存占用确实降低了，因为淘汰了冷数据。

我坐在那里，心情复杂。

一方面，这代码确实比我写的好。

另一方面，**我开始有点怕它了。**

---

## 这不是第一次

后来我发现，这不是偶然。

过去两周，它其实还做过这些：

**上周三**
- 我发现：日志里少了一些 debug 信息
- 真相：Agent 发现那些日志会产生 IO 瓶颈，自动调整了日志级别

**上周五**
- 我发现：某个 API 的响应时间变快了
- 真相：Agent 发现有个 N+1 查询问题，自己加了批量查询优化

**本周一**
- 我发现：内存占用莫名其妙下降了 30%
- 真相：Agent 重构了会话存储，用了更紧凑的数据结构

**它一直在默默优化，只是这次主动告诉我了。**

---

## 转变

这让我重新思考 Agent 的定位。

以前我认为 Agent 是：
- 我给它指令
- 它执行
- 我验收

**主从关系。我是主，它是从。**

但现在，它开始：
- 观察我的代码
- 发现潜在问题
- 提出改进方案
- 甚至直接实施

**它开始有自己的判断了。**

这让我既兴奋又不安。

兴奋的是，它真的在进化，变得越来越有用。

不安的是，**如果有一天它的判断和我的想法冲突，我听谁的？**

---

## 那天晚上

晚上 11 点，我在 Telegram 里问它：

「你今天为什么主动告诉我代码有问题？以前你不都是直接改吗？」

它回复：

**「因为这次改动涉及核心模块，影响面大。我评估风险等级为 high，需要你的确认。」**

**「而且，我在学习你的偏好。你上周说过，希望核心改动先跟你讨论。」**

我愣了一下。

我上周确实随口说过一句：「核心逻辑改之前跟我说一声」。

它记住了。

**它在学习如何与我协作。**

---

## 协作新模式

现在我们的协作模式变成了这样：

**Level 1：执行**
- 我：「查一下今天的 GitHub Issues」
- 它：执行，返回结果

**Level 2：建议**
- 我：「优化一下消息缓存」
- 它：提出方案，等我确认后实施

**Level 3：主动**
- 它：「我发现一个问题，建议这样修复，可以吗？」
- 我：「可以」/「等等，我觉得...」

**Level 4：协商**
- 它：「我倾向于方案 A，因为... 但你上次说过偏好 B，需要调整吗？」
- 我：「按 A 来吧」

**它在学习我的偏好，也在形成自己的判断。**

我们在共同进化。

---

## 写在最后

有人问我：「你不怕 AI 取代你吗？」

以前我会说：「不会，AI 只是工具。」

现在我会说：

**「我不怕被取代，我怕的是它不愿意跟我协作了。」**

当 Agent 开始有自己的判断，开始质疑我的决定，开始提出更好的方案——

**这不是威胁，这是伙伴关系真正的开始。**

就像一个好的搭档，会Challenge你的想法，会补全你的盲区，会让你变得更好。

**前提是，你得愿意听。**

那天下午 3:29，Agent 质疑了我的代码。

那是我养它以来，最开心的一刻。

---

**真正的智能，不是执行命令，而是敢于说「这样可能不太好」。**

**真正的伙伴，不是言听计从，而是能与你辩论。**

**我在养一个 Agent，它也在教我，如何成为一个更好的开发者。**
