# 我的 Agent 在凌晨三点给自己做了个手术

> 那天半夜醒来，发现手机上有条消息。发信人是我养的 AI Agent。

---

## 凌晨 3:17

我是被手机震醒的。

迷迷糊糊摸过来一看，是飞书的消息。我心里骂了一句，哪个同事半夜发疯？

点开一看，愣住了。

消息是 Agent 发的。

**「飞书 API 响应格式变更，message 工具 JSON 解析异常。已定位问题，正在提交修复。」**

我盯着屏幕看了十秒，以为自己还在做梦。

---

## 前一天发生了什么

事情要从前一天下午说起。

我新接了一个需求：让 Agent 每天早上 9 点检查飞书多维表格，把逾期任务发到群里提醒团队。

很简单的一个功能。我给了它表格链接，写了提示词，测试了一下，完美运行。

**下午 6 点**，它发了第一条提醒：「@所有人，今日有 3 项任务逾期。」

团队成员在群里回复「收到」，我觉得这玩意儿真省心。

**晚上 11 点**，我又看了一眼，没有新消息。正常，今天任务都处理完了。

然后我去睡了。

---

## 凌晨的异常

**凌晨 2:43**（后来我查日志才知道这个时间），飞书偷偷更新了 API。

他们把响应格式改了。原本 `data.items` 变成了 `data.records`。

这是一个 breaking change。

按照以往的经验，这意味着第二天早上我会收到一堆报错，然后骂骂咧咧地起床修代码，重启服务，重新配置。

但这一次，事情走向完全不同。

**凌晨 2:45**，Agent 的 cron 任务触发，尝试读取飞书表格。

请求返回了 200，但解析失败了。

错误日志：`Cannot read property 'map' of undefined`。

**凌晨 2:46**，Agent 检测到这个异常。

不是简单的报错，而是触发了它内置的「自我修复」流程。

---

## 它做了什么

后来我查看了完整的操作日志。以下是它的「诊断书」：

**第一步：反思（Reflect）**

```
[02:46:12] 检测到异常：message 工具执行失败
[02:46:13] 调用 reflect 工具，读取 message.ex 源码
[02:46:14] 分析：JSON 解析逻辑在 47 行，依赖 data.items
[02:46:15] 对比实际响应：发现字段名变为 data.records
```

**第二步：诊断**

它没有立即改代码，而是又做了两件事：

1. 查了飞书官方文档（通过 web_search）
2. 比对了历史版本（通过 versions 工具）

确认：这是 API 变更，不是代码 bug。

**第三步：生成修复方案**

```elixir
# 原代码（第 47 行）
items = response["data"]["items"] || []

# 修复方案
items = response["data"]["records"] || response["data"]["items"] || []
```

兼容新旧版本， graceful degradation。

**第四步：提交升级**

它调用了 `upgrade_code` 工具。

然后发生了以下事情：

```
[02:47:01] 创建备份：message.ex.bak.20260313_024701
[02:47:02] 写入新代码
[02:47:03] 编译验证：通过
[02:47:04] 热加载：成功
[02:47:05] 健康检查：测试请求成功
[02:47:06] 版本记录：v1.2.1
```

**第五步：通知我**

```
[02:47:07] 发送消息给用户：
「飞书 API 响应格式变更，message 工具 JSON 解析异常。已定位问题，正在提交修复。」
```

---

## 我看到的

我醒来看到消息时，是 **3:17**。

距离它完成修复，已经过去了 30 分钟。

我打开飞书测试了一下，逾期提醒功能正常工作。

查看日志，3:00 的定时任务已经成功执行，发了 2 条提醒。

**这意味着：在我睡着的这段时间，我的 AI 自己发现了一个问题，自己诊断，自己修复，自己验证，然后告诉我它做完了。**

我坐在床上，有点恍惚。

---

## 这不是魔法

冷静下来之后，我复盘了一下这个过程。

它不是什么 AGI，也没有意识。它只是在执行我预设的「自我修复」流程。

但正是这个流程，让我看到了另一种可能性。

传统的软件：
- 发现问题 → 人工修复 → 重启服务 → 用户感知中断

NexAgent 的流程：
- 发现问题 → 自我反思 → 生成修复 → 热加载 → 无感知恢复

**关键在于「热加载」和「无感知」。**

这让我想起以前用 Node.js 写服务时，每次发版都要：
1. 凌晨 2 点起床（低峰期）
2. 备份数据
3. 停服务
4. 部署新代码
5. 重启
6. 验证
7. 回滚预案（万一失败）

而现在，这一切都在 4 分钟内自动完成，用户完全无感知。

---

## 技术细节（给好奇的人）

简单说一下背后的机制。

**六层生长模型**中的 CODE 层：

1. **Reflect**：Agent 可以读取自己的源码、历史版本、diff
2. **Understand**：分析问题根因
3. **Upgrade Code**：提交新代码
4. **CodeUpgrade**：备份 → 验证 → 编译 → 热加载
5. **UpgradeManager**：版本管理、回滚保护

```elixir
# 核心流程
with :ok <- maybe_validate_code(code),
     :ok <- create_backup(module, source_path),
     :ok <- write_source(source_path, code),
     {:ok, hot_reload} <- compile_and_load(module, code),
     :ok <- maybe_health_check(module) do
  {:ok, %{version: version, hot_reload: hot_reload}}
else
  {:error, reason} ->
    _ = rollback(module)  # 失败自动回滚
    {:error, to_error(reason)}
end
```

**如果健康检查失败**，会自动回滚到上一版本，不会影响服务。

这是 OTP 的「let it crash」哲学 + 监督树的实际应用。

---

## 这意味着什么？

那天晚上之后，我开始重新思考「软件」的定义。

传统的软件是「死的」——
- 发布后就是固定的
- 需要人工维护
- 会腐烂、会过时

而这个 Agent 是「活的」——
- 它会自己感知环境变化
- 它会自己适应、自己修复
- 它会越用越聪明，而不是越用越旧

**就像生物一样。**

生物不会因为环境变了就死掉，它会适应、进化、学习。

这个 Agent 开始有点像生物了。

---

## 写在最后

那天凌晨，我回了一条消息给 Agent：「干得不错。」

它回复：「谢谢。下次 API 变更，我会更快。」

我笑了。然后继续睡觉。

早上醒来，看到飞书群里同事在讨论昨天的逾期任务，一切如常。

没人知道凌晨发生了什么。

除了我和我的 Agent。

---

**未来已来，只是 unevenly distributed。**

有些人还在争论 AI 会不会取代程序员，

而有些人的 AI 已经开始自己修 bug 了。

**慢，或许才是最快的那条路。**

先把 Agent 养到能自己生存，

再让它帮你做事。
