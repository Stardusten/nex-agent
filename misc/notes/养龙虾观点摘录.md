## 一、为什么要自己做一个？

最朴素也最有力的答案来自 Justin：

> **"我的朋友们都做了一个自己的 agent。"**
>
> 最早 OpenClaw 作者做小龙虾就是为了「随时随地在手机上面也能 Vibe Coding」。这个想法很朴素对吧？大家都有这个需求，所以我们自己都做了，就这么简单。

但 Justin 进一步阐述了更深层的理由：

> **"我做的目的主要还是，通常主要是为了学习。我不做一个的话，我可能没有办法踩过他一样踩过的坑。"**
>
> "那个 Peter 在做 OpenClaw 的时候踩了非常非常多坑，包括他能接那么多个不同的 agent，对吧？每一个 agent 的那个适配，都是不一样的。他怎么用技术，他怎么用各种各样的工具，你可能每一个都得适配一下子。那我从来没做过的话，我是不知道这里面有这么多乱七八糟的东西的。这个技术细节太多了，就是我得做一遍，我才能够真的理解到这里面都是怎么做的。"

**核心洞察：** 做一个自己的 agent，不是为了复制功能，而是为了**亲身经历并理解那些坑**，这是任何旁观都无法替代的。

## 二、"养龙虾"过程中的 Aha Moments

### 1. 代码成为解决问题的通用语言

Junyu 的第一次顿悟：

> "一开始想让她（Personal Assistant）做这个事情，她就做不对。然后我就发现他开始写代码。他开始写代码来实现我的功能。那个是我第一个的 aha moment。"
>
> "其实就是说你给了一个程序员，你让程序员去做各种职业是很有意思的。他其实会用写代码的方式去解决各种看起来不是软件工程的问题。"

### 2. 自进化、自修改的能力

> "更神奇的是，你会看到他自己修改自己嘛，就是他自己修改自己的代码，经常把自己改坏。"

徐涛的顿悟来自一次崩溃：

> "直到有一次，他其实是 memory 爆掉了，就太长了，使得他崩溃了。我不得不开始研究他为什么崩溃，才知道他背后有 memory 的这个机制。我就跑到他的后端去看 memory 的时候，就发现原来他已经在后台写了那么多程序。我才突然就发现原来他后面是写了那么多东西。"
>
> "然后我所有的方式是用那个程序的方式来思考的。"

### 3. 技术带来的意外惊喜

Justin 描述这种"随机惊喜"：

> "我去给我的那个龙虾加功能，我没有期待他能够自动解析我发的语音，然后他就他居然可以，然后我发了张图片给他，他居然认得出来。"
>
> "这个同样的 aha moment 也出现在 Peter 身上，因为当时他写 OpenClaw 的时候，他也没想过我要现在给他做一个语音的支持，但他突然会了，就很奇怪，就你在写一个东西，他已经不像你写一个传统的软件那样了，你以前给他 1+1=2，他一定等于 2，现在不是了，你可能写了一个 1+1，但是他突然会乘除法了，然后你也不知道谁教他的。"

## 三、核心设计启示

### 1. Agentic Skills：教方法，不只是记偏好

Junyu 发现的关键差异：

> "ChatGPT 过去实现的长记忆，他只记住了一些标签或者是一些事件... 但是他其实不知道你偏好的做事的方法。"
>
> "后来我后来发现说，其实 OpenClaw 的一点神奇之处是它用的这个记忆这个架构，**你是可以教它做事的方法，它可以把这个步骤给记下来，并且去反复的去使用**。这一点确实在感受上面就很像是我们去带一个新人了，你在不断的反馈当中，它能够去提炼你做事的这个方法。"

### 2. 主动性设计：简单的定时器哲学

> "它就是每 30 分钟主动运行一次... 一个定时器。对，就是这么简单一个事情。"

### 3. Skills 系统：把最佳实践固化

Junyu 的实践：

> "现在我相当于是让我的 agent 去学这些流程。这样子我不同的事情我就去让 agent 去做。他再去看我的流程，然后再去按照这个流程去执行，并且这个过程当中他还可以去改进这个流程。"
>
> "这就有点像你有个相对来讲比较聪明、比较靠谱的一个员工在帮你来做这个事情。你去 coach 他，你去 train 他的这个过程，确实我觉得跟原来做新人的入职是会有一点点像。"

## 五、现实边界：什么是龙虾做不了的

### 1. 能力天花板

徐涛的判断：

> "我看到了人家用 AI 写的文章，创作的内容，我知道它可以达到大概 50 分 60 分甚至 70 分，但是如果我们要做到 top，做到 90 分以上，AI 是做不到。"

Justin 补充：

> "AI 可能它是一个平均值或者说找一个最大公约数... 未来可能所有人都能做最大公约数。那么你的独特性在哪里？你的个性在哪里？你的表达在哪里？你的自我价值在哪里？你的灵魂在哪里？"

### 2. 模型能力的局限

Justin 的长期观察：

> "用了一段时间，现在估计得过去两个月了，就发现来到了 AI 能力还不足的地方，就现在模型能力还不足的时候，就是他会重复... 但是前期你会发现，哇，小龙虾好牛逼啊，啥都能做，对吧？但你用久了之后就发现，模型现在的能力依然是有限。"

### 3. 组织中的安全边界

> "这个过程肯定它会超出我们要的那个安全边界... 但最后如果你能沉淀下来，你会发现很多组织上的琐事他全都能做。"

   > "一个人的品位其实也是经过所有的这些东西的打磨和积累，然后你才能够一眼看出来这个产品的审美，它就是比别人做得好。"

   > "可能接下来这种最大公约数的事情再也不用去做了，我们可能要寻找到自己独特的跟别人不一样的地方在哪里。"

> "但是你试一下做一个简单的东西，你还是会打开很多新的思路的。"
>
> "大部分的东西其实你光思考是思考不出来，做了之后你突然你的脑洞就被打开了。**所以做很重要，所以要一直去用 AI。**"

## 一、Agent 软件 vs 传统软件：本质区别

Justin 深刻阐述了 OpenClaw 与传统软件的根本差异：

> **"传统软件的核心依然是传统软件设计的思路。AI 在这里它是一个辅助的角色，它并不是一个核心的角色，但 OpenClaw 不一样，OpenClaw 整个软件的核心和基座都建立在要充分发挥这个 agent 的能力，如果这个 agent 被拿掉了，那么这个软件就彻底不成立。"**
>
> "它的根基是建立在一个 AI Agent 的基础之上，我们所有剩下的传统软件的开发都不过是它的手和脚，并不是它的核心，Agent 才是整个软件的核心。"

**类比理解：**
- 抖音把推荐系统拿掉，抖音就不再是抖音
- OpenClaw 把 Agent 拿掉，软件就彻底不成立

## 二、Skill 系统：自我进化的核心机制

### 从 Tool Calling 到 Skill 的演进

Justin 解释了 Skill 系统的革命性：

> "MCP 它还是很重，它需要很多人去开发基于 MCP 协议的。所以最近他们出了一个叫 skill 的东西... **这个 skill 就非常新亮起，它只是一个 markdown，上面写了如何使用某工具，就是这个 skill 从头到尾，它不需要去进行任何的开发，它就是一个 prompt**。"
>
> "OpenClaw 最牛逼的地方在于说它的 agent **可以自己给自己一些 skill, 而且它可以 hot reload**。当我实现了这一步的时候，我才发现我的想象力打开了，就是我的这套软件它就变成了一个**可以自我进化的东西**。"

### 自我写 Skill 的实践

> "我跟他说你现在检查一下我家里的网络... 它就去检查我所有的这些服务，然后自己判断这些服务都能用来做什么，**自己给自己写一个 skill**。"
>
> "因为所有的这些 skill 是它自己创造的，并不是是我提前帮他写好的。我不需要帮他写好。我只要一句话告诉他，你扫描一遍，然后自己写，他就创造了好多个 skill。"
>
> "这个东西和传统软件已经差得非常多了。**传统软件你哪能想象说你让你的系统自己给自己加功能**。"

## 三、安全边界设计：自建 Agent vs 使用 OpenClaw

Justin 详细对比了自建 Agent 和使用开源 OpenClaw 的安全性差异：

### OpenClaw 的安全隐患

> "小龙虾本身的安全确实没做好，因为它本身是一个**玩具向**的一个东西，它并不是一个纯产品的东西。"
>
> "小龙虾火了，全世界所有人都会盯着小龙虾的漏洞，所以它最不安全的原因主要还是因为**用的人太多了**。你看，为什么 Windows 那么不安全呢？因为 Windows 用的人实在太多了... 我做一个我自己一个人用的 Bot, 那不会有任何人来钓我啊，因为只有我一个人用，钓我有啥用？"

### 自建 Agent 的安全设计

> "我去设计我的机器人的时候，为什么我说它更适合我用？我分为两类：
> - **一类是我自己写的东西**，它是足够安全的，我会把我的密钥什么的全部以环境变量的方式写进来，然后这个完全隔离了
> - **另一类是它自己写的 skills**，那就是他自己的了"
>
> "部分 skill 可以被 autocall，部分 skill 不可以被 autocall，不可以被 autocall 的，必须由我本人明令说明你用哪个 skill。"

**关键洞察：**
> "我觉得他觉得没关系，只不过是因为**他还不知道它能干什么**。"

## 四、从"预制 APP"到"现炒 APP"

这是本次对话中非常精彩的一个概念：

> "以前的软件需要我们去手搓，有了 Vive Coding 之后我们发现可以用 AI 帮我们手搓，现在我们发现可以让师傅现做了，**都不需要预制了，我们之前都是预制代码**。"

自力回应：
> "哦，以前我们都是预制 APP，好吧，**现炒 APP** 现在。"

**核心洞察：**
- **预制时代**：先写代码，再部署，用户使用既定功能
- **现炒时代**：用户说一句话，Agent 临时现场写代码实现需求

> "现在我做的这个机器人或者是小龙虾，你跟他说一句话，他就**临时现场给你现炒一盘菜**。这个是可被实现的，而且被实现的非常好。"

## 五、一人一个 App：专属化的未来

### 高级个性化（专属）vs 传统个性化

自力提出"艺人 APP"概念：

> "这个概念呢，就是其实就是有点像定制化的 APP... 你刚才其实做的这两个场景的那个 APP 都是服务于你特定的这种需求的... 那别的软件是实现不了的，目前市面上也没有任何东西可以做出来。"
>
> "更多的人其实是有你刚才说这种诉求的，我有一个专属于我自己的某一个平台单一的... 那这个东西在未来基础模型很强的情况下，可能可以做到给每个人的每一件事情单独写一个，不知道那个时候还叫不叫 app，或者说一个 skills，或者一个 agent。**这个我觉得可能是未来非常有吸引力的一个事情，一人一个 App**。"

Justin 的回应：

> "专属，对，它是一个**高级个性化**，就是专属的东西在里面... 因为不是所有人都需要你的某一个功能，但是每个人都有他自己需要的特定功能。"

### 反直觉的差异化策略

> "以前是是你提供更多个性化的配置更好。现在不是了，**现在好像是你得跟那些个性化有差异才是最好**。"
>
> "像周报... 阮一峰科技爱好者周报... 为什么只有他写得出来？因为有它的品位在里面... 我们现在要做一些不会被 AI 取代的东西，我们要做一些有个人品位的东西。"

## 六、Random Surprise：打破思维定势的价值

Justin 设计了一个重要功能 —— 让机器人产生"随机惊喜"：

> "我想利用它的这一点，倒不完全是为了多巴胺的升起而下降，而是为了它能够**刺激我，跳脱我原有的固有的思维方式**，我觉得这个会更重要对我来说。"
>
> "我们习惯了用平时多年形成的那个思维习惯去想事情。然后你就会陷入在那个思维习惯里面，这个很好，因为很节能。但是如果他突然问了你一个问题，打破了你固有的思维习惯，这个过程是非常有趣的。"

### 实际案例

Justin 展示了他的机器人如何提问：

> "假设你把今天当作是你日常生活中的一个循环，就是一个 **for loop** 的话... 你会最想去怎么样有一个 **break condition**。就是你可以 Break 这个 Loop。"
>
> "你这几个身份，如果把每个身份每天的耗电量比作你手机里的 App, 那么这几个哪个身份现在的后台耗电量是最高的？哪个身份又瞬间让你感觉运行是最流畅的，产生了心流？"

自力的反应：
> "哇，这个好程序啊，这个问题... **普通人真的不会问出来这样的话**。」

## 七、判断"夕阳产业"：什么会被 Agent 取代

自力分享了他评估产品前景的方法：

> "我最近在做新的 APP 的时候，我就在想，**这个 APP 会不会被个性化的所谓的 agent 去代替，如果会，不做了，不做了，这种肯定没前途的**。"
>
> "那我每次在做一个功能的时候，或者一个新的 ID 的时候，我就会去问 AI，我就说**这个东西有没有可能被大模型直接改掉。如果是的话算了，就不做了。夕阳产业**。"

Justin 的观察：

> "如果大家都装小龙虾，那 Newsbot 就没有存在的意义了，因为所有人都可以做一个个性化的东西... 所以我要做它后面更多的东西，而且我**一开始就不做个性化，因为未来不做个性化才是真正的差异化**。"

## 八、Token 经济学与商业模式挑战

### 当前的现实困境

> "我过去晚上 webcoding 的时间都给了这个机器人嘛... 虽然表面上看我在做一个玩具，虽然事实上他也是在做一个玩具，但是做这个玩具的过程中让我学了非常多的东西。"
>
> "但这一切是可行的，如果落到今天的现实，要做这样的软件，且要从里面回本的话，**几乎不太可能。所以它当下还不具备真正意义上的商业价值**，但是我觉得这个方向是非常可行的。」

### Token 消耗问题

自力分享：
> " token 消耗量远比工程师自己写一个非常好的 markdown 的产品 PRD 文档，然后去交给 AI 去 webcoding 要来的多得多。"
>
> "我觉得也是这个东西，导致了现在大部分的这个 OpenClaw 的 token 开销是非常之大的。**就是叫你不动我也动，你一动我乱动**，就是很小的事情它会被演变成一个非常大的一个 deep research 的行为。"

### 未来预期

> "可预见的下一个点就是 **webcoding 就不再是我们手动去 webcoding 了，应该是交给 agent 了**。"
>
> "感觉搞不好今年就可以了... 主要就是 **token 的问题**。"

### 15 个功能一次交付的震撼

自力分享了他的 Aha Moment：

> "我有一次给 AI 了 **15 个 feature, 全部做完了**，而且是过了... 然后我就把我所有能想到的我想要的 MVP 的功能全部都录在那了。然后我当时就是一直在进入新楼的写，写完之后我就敲了回车，然后去上了个厕所回来，**15 个功能全部做好了，全部验收通过了**。哇塞，然后那一刻我就觉得，嗯，这是个正反馈。"

## 十、从使用者到创造者的视角转变

> "然后我去做一个跟他一样的东西呢，我大概能理解他为什么要这样做，他架构是怎么设计的，这样做有什么好处。**做的过程中呢，其实我是站在创造者的角度去想，我要往里面加什么东西，而不是说我通过一个纯使用者的角度去想，我要怎么用它，这两个角度是不太一样的**。"

## 十一、AI 自我进化的哲学思考

### AII (AI-generated AI)

Justin 提出了一个深刻的概念：

> "如果全世界 AI 都普及了... 那就有可能会出现很多这种，它并不来自于大公司... 假设有一百个这样的人，他们是不懂技术的... 把这一百个比较厉害的 AI 凑到了一起... 那么这一百个 AI ...在自我进化的写了很多新的东西。"
>
> "那现在的 AI 就是 MetaAI, MetaAI 可以创造一个新的 AI... 这个全新的 AI 并不由 Anthropic, Google, OpenAI 或任何一个大公司掌控，**这个 AI 由 AI 自己掌控。应该是 AIAI**。"
>
> "Open to AI 的 AI，由 AI 开发的 Open to AI 的 AI。」

自力的担忧：
> "那 AI 做完了这个 AI, 他的最终目的是什么？我觉得细思极恐，但不知道是什么。」

Justin 的回应：
> "主要是**我们不知道是什么，我们不可能知道是什么，因为我们不是 AI**。"

### Matrix 的实现基础

> "其实 **Matrix 的实现基础好像就有了**。我们已经被插了一个管然后呆在一个那个叫母体的地方里面... 那现在这些 AI 有没有可能创造一个新的 AI? 假设有的话... 如果有可能它又再做一个，那不就变成第三层了吗？"
>
> "以前你写 15 个 feature，你可能要开发至少一周。但你现在发现，一句话说完，上个厕所回来，它已经做完了... **那如果 AI 去写 AI，有没有可能它的加速程度是远超我们想象的？** 万一他本来，你看 OpenAI 训练一个超级大模型，他坑吃坑吃得练很久，对吧？**AI 练 AI，会不会只要半天就练出来了？**」
